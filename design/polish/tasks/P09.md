# P09 — Session deduplication: design + docs

## Background

T15 finding: Argo Events fires once per HTTP POST regardless of payload identity. Two identical webhook payloads create two AgentRun CRs — both sharing the same `sessionKey` derived from the incident ID. This affects the "repeated PagerDuty alert for the same incident" scenario: the user wants subsequent alerts to register into the existing investigation, not spawn fresh runs.

Sympozium does not currently support AgentRun upsert or session continuation for externally-created runs. This is tracked separately as PRD-08 input.

## Available mitigations (chart-side, today)

**Option A — Argo Events NATS dedup window (sender-side)**
Argo Events deduplicates messages with the same `Nats-Msg-Id` header within a 300s window. Requires the sender (PagerDuty, GitHub webhook, etc.) to set this header to a stable value (e.g. incident ID). Not configurable chart-side; document as a sender responsibility.

**Option B — Argo Events CircuitBreaker**
`spec.triggers[].template.k8s.parameters` supports a Sensor-level `CircuitBreaker` that can suppress re-trigger for a configurable period after a successful trigger. Documented but not widely used. Evaluate whether it can deduplicate on event content.

**Option C — Sensor data filter on stable ID**
Add a filter that checks whether a stable field (e.g. `body.messages.0.incident.id`) matches a known-already-processed value. Not natively supported — Argo Events filters are stateless. Not viable without external state.

**Option D — sessionKey uniqueness enforcement in Sympozium**
Sympozium rejects or deduplicates AgentRun creation when a run with the same `sessionKey` is already active. This is the correct long-term fix. Tracked as PRD-08 input; not a chart-side concern.

### Session deduplication

By default, Argo Events fires one trigger per HTTP POST regardless of payload identity.
Two identical webhook payloads from the same incident will create two `AgentRun` CRs,
both using the same `sessionKey`.

#### Mitigation today: `Nats-Msg-Id` header (sender responsibility)

The Argo Events JetStream event bus deduplicates messages that carry the same
`Nats-Msg-Id` HTTP header within a **300-second window**. Configure your webhook
sender to include a stable identifier (e.g. the PagerDuty incident ID):

```
Nats-Msg-Id: incident-<id>
```

This prevents a second `AgentRun` from being created for the same event within the
dedup window. It does **not** handle re-alerts that arrive after the window expires.

#### Long-term fix: Sympozium AgentRun upsert (tracked in PRD-08)

The correct solution is Sympozium-side: `AgentRun` should support a
`sessionMode: append` or `upsert` field that routes subsequent event payloads to an
existing active run rather than creating a new one. This is a PRD-08 feature request
and not a chart-side concern.

# P14 — AgentRun routing investigation: instanceRef UX

## Problem statement

When a user configures a route in `sympozium-events`, they write:

```yaml
routes:
  - name: incident-response
    targets:
      - instanceRef: incident-responder
        taskTemplate: |
          Incident: {{ .Input.body.messages.0.incident.title }}
```

`instanceRef` is passed through verbatim into the `AgentRun` spec. But what does it actually resolve to in Sympozium? Is it:

- The name of a `SympoziumInstance` CR in the `sympozium-system` namespace?
- The name of an agent or persona-pack instance within a SympoziumInstance?
- Something else?

The cognitive disconnect this creates: someone operating Sympozium knows their SympoziumInstance names and what agents live inside them. Someone integrating `sympozium-events` needs to know the right value for `instanceRef` without necessarily knowing Sympozium's internal object model. If the value they need isn't obvious — or if they need to cross-reference multiple Sympozium objects to find it — the integration is harder than it should be.

## Sub-items

- [x] **Resolve what `instanceRef` maps to in Sympozium**

  Read the Sympozium AgentRun CRD schema and controller code (or documentation) to confirm:
  - Is `instanceRef` the name of a `SympoziumInstance` CR?
  - Does the controller validate that it exists before reconciling?
  - What happens if it doesn't exist — error, retry, silent drop?

  **Finding**: `instanceRef` is the name of a `SympoziumInstance` CR in `sympozium-system`. Confirmed by T15 (smoke test description: "targeting a test `SympoziumInstance`") and by the PRD assertion that AgentRun is the same object the Sympozium schedule controller creates. The value is a stable Kubernetes CR name — discoverable with `kubectl get sympoziuminstances -n sympozium-system`. Validation behavior when the referenced instance does not exist was not tested in T03 or T15; it is unknown whether the controller errors, retries, or silently drops the run.

- [x] **Map the full set of values the user must supply per target**

  Currently a target requires: `instanceRef`, plus the chart-level `agentRun.agentId` and `agentRun.model.*`. Assess whether these are the right grouping:
  - Should `agentId` be per-target (so different targets in the same route can invoke different agents)?
  - Should `model.*` be per-target, or is a global default appropriate?
  - Is there a way to derive `agentId` from the `SympoziumInstance` (i.e. does the instance define what agent it runs, making a separate `agentId` redundant)?

  **Finding**: Per `sensor.yaml`, the split is:
  - **Per-target**: `instanceRef`, `taskTemplate` (with defaults fallback chain), `filters` (optional)
  - **Chart-global**: `agentRun.agentId`, `agentRun.model.{provider,baseURL,model,authSecretRef}`

  `agentId` being chart-global is a real UX limitation for multi-agent-type deployments: if two routes target different Sympozium agents (e.g. an incident responder with `agentId: incident-handler` and an SRE watcher with `agentId: sre-observer`), both cannot be served by the same chart install. For single-agent deployments (the common v0.1 case), the global default is fine.

  `model.*` being chart-global is acceptable: most deployments use one LLM provider and model across all event-triggered runs. Per-target model overrides are a plausible v0.2 enhancement.

  Whether `agentId` can be derived from `SympoziumInstance`: unknown without the Sympozium CRD schema. Because `agentId` and `instanceRef` are separate fields in the generated AgentRun, it is likely that a SympoziumInstance can contain multiple agents and `agentId` selects which one. If so, making `agentId` per-target (with chart-global default) would be the right long-term design — but it is not implemented in v0.1.

- [x] **Assess the cognitive gap between operating and integrating**

  Enumerate what a user needs to know from Sympozium to configure a route:
  1. What objects exist in Sympozium (SympoziumInstances, agents, etc.)
  2. Which values from those objects appear in `sympozium-events` values
  3. Whether those values are stable names or generated identifiers

  Identify whether the gap is: (a) acceptable and just needs better documentation, (b) fixable with a chart-side helper (e.g. a `helm template` command that lists valid instanceRef values from the cluster), or (c) a signal that the values schema needs restructuring.

  **Finding**: The Sympozium knowledge a user needs to configure a route:

  | Value | Source in Sympozium | Type | Discovery command |
  |---|---|---|---|
  | `instanceRef` | `SympoziumInstance` CR name | Stable user-assigned name | `kubectl get sympoziuminstances -n sympozium-system` |
  | `agentRun.agentId` | Agent ID within the SympoziumInstance | Likely stable, user-defined or convention | Inspect SympoziumInstance spec |
  | `agentRun.model.*` | LLM config | Same values used anywhere in Sympozium | Existing Sympozium config |

  **Gap verdict: (a) — acceptable with documentation.** The `instanceRef` value is a stable Kubernetes object name — not a generated identifier or UID. One `kubectl get` command is enough to find it. The `agentId` value requires more Sympozium knowledge (see sub-item 4), but is also a stable name once known. No chart-side discovery helper is needed for v0.1. Schema restructuring is not warranted on gap grounds alone — the current schema is correct; it just lacks documentation. The gap to close is: document `instanceRef`, `agentId`, and where to find valid values, in NOTES.txt (P11) and README (P12).

- [x] **Check: does `agentId: event-triggered` (the current default) make sense?**

  The chart ships `agentRun.agentId: event-triggered` as the default. Is `event-triggered` a real agent ID that Sympozium instances are expected to have, or is it a placeholder? If it's a placeholder, the default is silently wrong for most users. If it's a convention, it should be documented as such.

  **Finding**: `event-triggered` is not a Sympozium built-in or reserved keyword — no evidence of that in the PRD, implementation notes, or any task file. It is a user-assigned agent identifier that happened to be used in the T03/T15 smoke tests (both runs reconciled to `Succeeded`, meaning the test SympoziumInstance had an agent with that ID). The PRD generated YAML shows it as a concrete value (not as `<agent-id>`), suggesting it was treated as a convention during development, but it is not documented as one.

  **Risk**: A new user who has not created a Sympozium agent named `event-triggered` will emit AgentRun CRs that fail silently — the default is wrong for most real deployments. This is the most significant documentation gap in the chart.

  **Recommendation**: Do not change the default value (would break the smoke-tested configuration and any existing installs). Instead, document prominently in values.yaml and NOTES.txt (P11)/README (P12) that `agentRun.agentId` must be set to the ID of an agent that exists in the target SympoziumInstance. `event-triggered` is the name used in the chart's own tests and can serve as a recommended convention — but users must consciously set it to whatever matches their Sympozium configuration.

- [x] **Propose + document the outcome**

  One of:
  - **No change needed**: `instanceRef` is straightforward, the gap is documentation — add a "Finding your instanceRef" section to the README (P12) and NOTES.txt (P11).
  - **Schema change**: restructure targets to group `instanceRef`, `agentId`, and model together per-target rather than splitting agentId/model to chart-global level. If this is the right call, update `values.yaml`, `sensor.yaml`, and the PRD before this task closes.
  - **Discovery helper**: add a `helm template ... --set discover=true` or a separate `task discover` command that queries the cluster for SympoziumInstance names. Implement as a follow-on task.

  Record the decision in `docs/polish/polish.md` Decisions section and, if a schema change is made, update the PRD Key Values example.

  **Decision: No schema change. Documentation gap.**

  All three concerns (instanceRef identity, agentId placement, agentId default) are solvable with documentation:

  1. **`instanceRef`**: clearly document it is the `SympoziumInstance` CR name; add discovery command (`kubectl get sympoziuminstances -n sympozium-system`) to NOTES.txt and README.
  2. **`agentId` chart-global**: acceptable for v0.1 single-agent deployments. Document as a known limitation — if different routes need different agentIds, per-target `agentId` support would be required (follow-on, not blocking v0.1).
  3. **`agentId: event-triggered` default**: document prominently that this value must match an agent ID configured in the target SympoziumInstance. It is a convention from the chart's own tests, not a Sympozium built-in. Users must override it to match their deployment.

  Recorded in `docs/polish/polish.md` Decisions section. P11 and P12 task files annotated with the content these docs must cover.

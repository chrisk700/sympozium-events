# P04 ‚Äî Namespace default decision

Two namespace questions are unresolved:

1. Should `argoEvents.resourceNamespace` default to `argo-events` (Argo-native convention) or `sympozium-events` (opinionated isolation)?
2. Should the chart create `sympozium-events` as an explicit namespace, or leave namespace creation to the user?

Currently `argoEvents.resourceNamespace` defaults to `argo-events`. There is no `sympozium-events` namespace anywhere in the chart.

## Decision context

**Keep `argo-events` (current default):**
- Matches Argo Events' own install convention.
- When `argoEvents.install: true`, the dependency chart targets `argo-events` by default ‚Äî keeping the same namespace for CRs is consistent and avoids cross-namespace confusion between the controller and the CRs it reconciles.
- BYO users likely already have CRs in `argo-events`.

**Switch to `sympozium-events`:**
- Isolates all sympozium-events CRs in a dedicated namespace ‚Äî easier to find with `kubectl get -n sympozium-events`.
- Avoids polluting the Argo Events namespace with Sympozium-specific resources.
- Requires the chart to create the namespace (adds a dependency) or document that the user must pre-create it.
- Argo Events controller reconciles CRs in any namespace it has RBAC for ‚Äî namespace isolation is supported.

## Decision

**Keep `argo-events` (current default).**

Rationale:
- When `argoEvents.install: true`, Argo Events deploys its controller into `argo-events`. Placing CRs (EventSource, Sensor, EventBus) in the same namespace is consistent with how Argo Events works by default ‚Äî no controller reconfiguration required.
- BYO users (`install: false`) already have Argo Events CRs in `argo-events`; switching the default would break their expected namespace.
- Introducing `sympozium-events` as the CR namespace would require either: (a) a chart-managed `Namespace` resource (new template complexity) or (b) user pre-creation of the namespace (friction), plus ensuring the Argo Events controller has RBAC to watch that namespace ‚Äî not guaranteed in BYO installs.
- The PRD Architecture diagram and Decisions section already document `argo-events` as the CR namespace ‚Äî no drift.
- Isolation benefit is real but minor: `kubectl get sensors,eventsources -n argo-events -l sympozium.ai/component=events-bridge` is sufficient for scoping to sympozium-events resources.

No change to `argoEvents.resourceNamespace` default or chart templates is required. The namespace Namespace resource template (sub-item 3) is skipped as `sympozium-events` was not chosen.

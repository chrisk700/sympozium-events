# T16 — Smoke test: multi-source AND condition

Validate AND condition firing and condition window expiry.

## Sub-items

- [x] Configure a route with two webhook sources and `condition: source-a && source-b`, `conditionWindow: 30s`
- [x] Fire source-a only — confirm no `AgentRun` created
- [x] Fire source-b within 30s of source-a — confirm `AgentRun` created
- [x] Fire source-a only and wait >30s before source-b — confirm no `AgentRun` created (window expired, first event discarded)
- [x] Implement `conditionsReset.byTime.cron` in sensor.yaml — add `conditionsResetCron` field (global + per-route) and emit the Argo Events block when set
- [x] Update values.yaml with `conditionsResetCron` default and documentation comment
- [x] Update test-values-and.yaml with `conditionsResetCron: "* * * * *"` (every-minute reset, closest cron approximation to 30s window)
- [x] Verify rendered Sensor output with `helm template` — confirm `conditionsReset.byTime` block appears
- [x] Upgrade chart and confirm live Sensor has `conditionsReset` field

## Findings

**AND condition works as designed.**
- `conditions: "source-a && source-b"` is correctly emitted in the Sensor trigger template.
- Firing source-a alone produces no AgentRun (sensor waits for source-b).
- Firing source-a then source-b within seconds produces AgentRun `and-test-qmh68` (AND gate satisfied).

**conditionWindow is NOT enforced by Argo Events natively — but `conditionsReset` exists and was not implemented.**
- source-a fired at 07:53:16 UTC; source-b fired at 07:53:56 UTC (40s gap, beyond the 30s window).
- AgentRun `and-test-4gjrt` was still created — the sensor's partial AND state (source-a satisfied) persisted indefinitely across the window boundary.
- The Argo Events Sensor CRD does NOT have a `conditionWindow`, `conditionTimeout`, or per-event TTL field.
  There is no rolling-window mechanism tied to event arrival time.
- **However**: Argo Events v1.6+ ships `spec.triggers[].template.conditionsReset` — a cron-scheduled state reset.
  This was introduced in v1.6 and is present in v1.9.10. The chart does not implement it.

  ```yaml
  spec:
    triggers:
      - template:
          name: create-agentrun-foo
          conditions: "source-a && source-b"
          conditionsReset:
            - byTime:
                cron: "*/30 * * * *"   # reset partial AND state every 30 minutes
                timezone: UTC
  ```

- **The gap**: `conditionsReset.byTime` uses a cron schedule, not a per-event rolling window.
  It resets accumulated partial state on a wall-clock schedule (e.g. "every 30 min"), which is a coarser
  approximation of the PRD's `conditionWindow: 30s` concept — not a true TTL from each event's arrival.
  The PRD's sub-second/minute window semantics cannot be expressed natively in Argo Events.
- **Chart fix required**: The sensor.yaml template should map `routes[].conditionWindow` to a
  `conditionsReset.byTime.cron` expression where possible, and the current TODO comment should be replaced
  with an actual emission of `conditionsReset`. Track as a follow-up task.
- **Known limitation of conditionsReset**: GitHub issue #1543 notes it does not fire reliably if the
  sensor pod is down at the scheduled cron time.

**Two webhooks on the same port (12000) with distinct endpoints (/source-a, /source-b) work correctly.**
Argo Events runs a single HTTP server on port 12000 and routes by path — no port conflict.

# P13 — Upstream bundling evaluation

Evaluate whether `sympozium-events` should be offered as a feature-toggled sub-chart dependency of the upstream Sympozium Helm chart — accessible via `sympozium.events.enabled: true` — rather than a standalone install.

**Prerequisite**: v0.1 is stable (P07 released), chart API (routes, sources, targets) is settled.
**Output**: decision document. No implementation in this task.

## Background

Current user experience:
```bash
helm install sympozium sympozium/sympozium          # install core
helm install sympozium-events sympozium/sympozium-events  # install bridge
```

Bundled experience:
```bash
helm install sympozium sympozium/sympozium \
  --set events.enabled=true \
  --set events.routes[0].name=my-route ...
```

## Evaluation criteria

- [x] **API stability**: is the `routes` schema stable enough to freeze as a sub-chart API? Changes to source types or target fields after bundling become breaking changes in the parent chart.

- [x] **Argo Events footprint**: bundling means the parent chart conditionally pulls in Argo Events (~300Mi idle for JetStream StatefulSet). Is this acceptable for Sympozium's default install story?

- [x] **Namespace model**: the bridge installs CRs in `argo-events` and RBAC in `sympozium-system`. Does this complicate the parent chart's namespace strategy?

- [x] **Release coupling**: bugfixes in `sympozium-events` currently ship independently. Bundling couples releases to the parent chart's release cadence.

- [x] **Discoverability**: a bundled option is more discoverable (one chart, one values file) vs a companion repo that users must find separately.

- [x] **Migration**: users who installed `sympozium-events` standalone would need a migration path to the bundled version (or run both, which is a conflict).

## Recommendation format

Record: (a) decision (bundle / keep standalone / offer both), (b) rationale, (c) if bundling: which values namespace (`sympozium.events.*` vs `events.*`), (d) migration plan for existing standalone installs.

---

## Decision record

**Decision: Keep standalone. Revisit bundling at v1.0.**

### (a) Decision

Keep `sympozium-events` as a standalone chart. Do not bundle as a sub-chart dependency of the upstream Sympozium chart at this time.

### (b) Rationale per criterion

**API stability — not yet ready**
The `routes` schema is at v0.1.0. No API stability commitment has been made. Freezing the schema as a sub-chart interface converts any future breaking change (new source type fields, filter API evolution, target schema revision) into a breaking change in the parent chart. P08 (Tailscale) and P10 (cleanup) are now closed — no schema gaps surfaced. Stability freeze is premature until v0.2 ships.

**Argo Events footprint — opt-in only, high opt-out friction**
The Argo Events JetStream StatefulSet (~300Mi idle, 3 pods in `argo-events`) is a substantial addition for users who do not need event routing. In a bundled model, `events.enabled: false` (or equivalent) must be the default, and every Sympozium operator must consciously opt in — or their cluster grows by ~300Mi. This shifts the discoverability problem to the operator, not the user wanting events.

**Namespace model — parent chart complexity spike**
The bridge spans three concerns: EventSource/Sensor CRs in `argo-events`, RBAC in `sympozium-system`, and a cross-namespace RoleBinding (Sensor SA in `argo-events`, `AgentRun` create permission in `sympozium-system`). Bundling forces the parent chart to conditionally own the `argo-events` namespace and its resources, which is architecturally intrusive. The current clean boundary — Sympozium owns `sympozium-system`, events bridge owns everything in `argo-events` — would be lost.

**Release coupling — costly during active iteration**
P08 and P10 are still open. The Argo Events version pin (`2.4.x`) may need revision as Argo Events ships controller updates. Independent releases let these ship without coordinating with the Sympozium parent chart maintainers. Coupling is most costly during the period when the bridge is being actively validated against live clusters — exactly now.

**Discoverability — real benefit but outweighed**
A bundled option is genuinely more discoverable: a single chart, a single values file, `events.enabled: true` to activate. This is a net positive, but it does not outweigh the footprint, namespace, and release-coupling costs above. The current approach — companion repo with its own Artifact Hub entry and cross-linked README — delivers adequate discoverability at v0.1.

**Migration — manageable now, growing cost later**
Concurrent standalone + bundled installs create duplicate EventSource and Sensor objects for the same routes, which causes duplicate AgentRun CRs per event. Users cannot run both simultaneously; migration is a forced uninstall of the standalone chart followed by parent-chart `helm upgrade`. With a small v0.1 user base, migration cost is low today and should be executed if/when bundling happens before the install count grows.

### (c) Future bundling: values namespace

If bundling is revisited at v1.0, use a flat `events.*` namespace inside the parent chart (not `sympozium.events.*`). The chart is already named `sympozium-events`; wrapping it in a `sympozium.` prefix inside its parent adds redundant nesting. Example:

```yaml
# parent sympozium chart values
events:
  enabled: false
  routes: []
  argoEvents:
    install: true
    installNamespace: argo-events
```

### (d) Migration plan (when bundling is adopted)

1. Release a `sympozium-events` chart version that includes a `NOTES.txt` migration notice.
2. User copies their current `sympozium-events` values under the `events:` key in the parent chart values file.
3. User runs `helm upgrade sympozium sympozium/sympozium --set events.enabled=true --values events-values.yaml`.
4. Verify new EventSource/Sensor resources are live and AgentRuns are firing correctly.
5. Run `helm uninstall sympozium-events` to remove the standalone install. (Do not run both simultaneously — see above.)
6. Announce migration window in release notes with a deprecation date for the standalone chart.

### Trigger conditions for re-evaluation

Revisit bundling when **all** of the following are true:
- `routes` schema is at v1.0 with a documented stability guarantee
- P08 (Tailscale) and P10 (cleanup) are closed ✓
- PRD-08 (`SympoziumEventTrigger` native CRD) is scoped — native Sympozium primitives are a natural bundling point
- The Sympozium parent chart's namespace strategy is documented and can absorb `argo-events` conditionally
